1. Executive Summary & Vision
🎯 ONE-LINE PITCH:  InternConnect is an AI-powered two-sided marketplace that connects ambitious students with startups, companies, and organisations through intelligent matching, verified portfolios, and flexible micro-internships.
1.1 The Problem
The internship market is broken on both sides:
Students spend weeks sending cold applications into black holes, getting no feedback, no transparency, and no way to stand out beyond a CV.
Companies — especially startups — cannot afford 6-month full-time internships. They need scoped, flexible talent for specific projects but have no efficient channel to find it.
Generic job boards (LinkedIn, Internshala) treat internships like junior full-time jobs. There is no differentiation by project type, duration, culture fit, or skill verification.
1.2 The Solution
InternConnect solves this through three core innovations:
AI Matching Engine — Scores compatibility across 40+ signals (skills, availability, work style, culture, project type) rather than keyword search.
Micro-Internship Format — 2–8 week project-based engagements unlock startups and SMEs as hiring companies, dramatically expanding the supply side.
Verified Portfolio System — Every completed internship piece is company-endorsed and displayed as proof-of-work, not just a line on a CV.
1.3 Vision Statement
🌟 VISION:  To become the definitive internship infrastructure for Asia — the platform where every ambitious student finds their first meaningful work, and every company finds the right talent at the right time.

6. Feature Modules — Detailed Specifications
6.1 Authentication & Onboarding
Single entry point for both user types. Role is selected on the signup screen and cannot be changed.
Screens
Login — Email + password. Social login via Google OAuth and GitHub OAuth.
Signup — Role toggle (Student / Company). Dynamic form fields based on role selection.
Email verification — OTP sent on registration. Must verify before accessing dashboard.
Forgot password — Email-based reset with time-limited token (15 minutes).
Profile completion prompt — Banner in dashboard until profile reaches 80% completion.
Functional Requirements
JWT-based authentication with refresh tokens (7-day expiry).
Google OAuth 2.0 and GitHub OAuth integration. Auto-populate name + email from OAuth.
Role selection cannot be changed post-registration. Separate accounts required for dual roles.
Rate limiting: max 5 failed login attempts before 15-minute lockout.
Session timeout after 30 minutes of inactivity. Silent refresh for active sessions.
On first login after signup, redirect student to profile builder wizard.
On first login, redirect company to role creation flow.

Job Card
Each card displays: company logo/initials, role title, company name + location, tags (remote/micro/paid + skills), AI match score badge, stipend, duration, Apply button, Save button.
"Featured" badge for sponsored/promoted listings. Card click opens role detail page.
Search Behaviour
Full-text search across role title, company name, skill tags, and description.
Debounced 250ms — no search button required, results update on type.
URL query params updated on each filter change — shareable/bookmarkable search state.
Empty state shows relevant suggestions when no results match.
Save role button persists to user's saved list without navigating away.
Sort options: Best Match (default), Newest, Stipend High-Low, Duration Short-First.
"Load More" pagination (no infinite scroll — reduces cognitive overload).

6.10 Portfolio & Endorsement System
The verified portfolio is InternConnect's strongest differentiator. Every piece of work completed through the platform can be company-endorsed. This section describes both self-uploaded and verified pieces.
Self-Uploaded Projects
Add via three methods: (1) file upload, (2) external link, (3) Figma share URL
Each project: title, description, type, year, skills demonstrated, cover image
Featured toggle — pins project to top of portfolio grid
Portfolio strength score (0–100) updates as projects are added
Public portfolio URL: internconnect.io/u/[username] (indexable by search engines)
Verified Internship Work
Created automatically when a company marks a deliverable as complete
Company endorses the piece, which adds a "Verified by [Company]" badge
Verified pieces show: company logo, intern's role, date, deliverable description, skills demonstrated

Recruiters cannot remove or edit verified pieces (integrity guarantee)

Companies endorse specific skills after project completion (e.g. "Figma — Endorsed by Stripe")
Endorsement level: 1–5 dots (set by company based on demonstrated proficiency)

View count per piece (from recruiters + other visitors)
Total portfolio views (weekly trend in dashboard)
Which companies viewed which projects (when viewing from company login)
6.11 AI Matching Engine
⚠️ NOTE:  This section describes the product-level specification. The actual ML model architecture is a separate technical document. This covers what the engine does, not how it is built.
What It Matches
For each student–role pair, the engine produces a compatibility score from 0–100%. This score is shown to both the student (as their match percentage) and the company (as the candidate ranking).
Input Signals (40+)
Category
Signals
Skills
Technical skill overlap, proficiency level match, tool-specific experience
Availability
Duration preference match, hours/week overlap, start date alignment
Domain
Field-of-study alignment, project domain history, expressed domain preference
Work Style
Remote/hybrid/on-site preference, communication style indicators
Culture
Company size preference vs actual company size, mission alignment signals
Past Activity
Portfolio project types, previous internship domains, saved company types
Application Quality
Profile completion %, portfolio strength score, endorsement count
Match Score Display Rules
90–100%: "Excellent Match" — green badge, shown first in feed
75–89%: "Good Match" — blue badge
60–74%: "Fair Match" — amber badge
Below 60%: No badge shown. Role still appears in search if filters match.
Companies see ranked candidate list sorted by score (highest first by default).

Icon colour should match surrounding text colour or use semantic colour (green for success, amber for warning).
7.9 Motion & Animation
Page transitions: AOS library, duration 700ms, ease-out-cubic, once: true
Hover transitions: 200ms ease (fast, responsive)
Panel/modal transitions: 350ms cubic-bezier(0.4, 0, 0.2, 1)
Scroll-triggered elements: fade-up with 60px offset
Hero orbs: slow float animation (8s, ease-in-out, infinite)

8. UX Principles & Interaction Patterns
8.1 Core UX Principles
Principle 1: Zero Dead Ends
Every empty state, error state, and dead end must have a clear next action. Empty search results show suggested roles. Empty application list shows "Browse Roles" CTA. A 404 page explains what happened and links back to homepage.
Principle 2: Progressive Disclosure
Never overwhelm users with all information at once. Company profiles use "Read More" expand. Role requirements use "Must Have" (visible) + "Nice to Have" (below fold). Advanced filters are collapsed by default on mobile. Forms show one logical group at a time in wizard flows.
Principle 3: Optimistic UI
UI state updates immediately on user action. Network call happens in background. If the call fails, revert the state and show an inline error — not a page reload. Save role: bookmark icon fills immediately. Shortlist: row turns green immediately.
Principle 4: Contextual Help
Every form field with a non-obvious purpose has a helper text or tooltip. No reliance on placeholder text as labels — labels and placeholders are always separate. Character counters on all text inputs with limits (appear at 80% of max, turn amber at 90%).
Principle 5: Mobile-First Decision Making
All layout decisions start from mobile. Desktop gets more space, not more features. Touch targets minimum 44×44px. No hover-only interactions for key features. Bottom navigation for key student actions on mobile.
8.2 Navigation Patterns
Desktop: top navigation bar (fixed, blurs on scroll with backdrop-filter). Scrolled state adds darker background and bottom border.
Mobile: hamburger menu opens a slide-down panel with all nav links.
Dashboard: left sidebar with section groupings. On mobile, sidebar collapses to icon-only or hidden with bottom nav.
Breadcrumbs on role detail and company profile pages for wayfinding.
Back navigation: always a visible "Back" link/button, never rely on browser back.
Active nav item: indigo left border + translucent background (sidebar) or underline (top nav).
8.3 Form UX Standards
Label always above input (never floating labels — they obscure input state).
Required fields marked with a subtle asterisk (*) in the label.
Validation: on blur for individual fields, on submit for form-level errors.
Error messages: specific and actionable. Not "Invalid input" — say "Email must include @".
Success: green border briefly on valid input, then remove to avoid visual noise.
Multi-step forms: show step number, total steps, and allow back navigation.
Auto-save: all multi-step form data saved to localStorage on each step advance.
Keyboard navigation: all interactive elements reachable by Tab. Focus rings visible.
8.4 Loading & Skeleton States
Skeleton screens (not spinners) for cards, tables, and lists.
Skeleton matches the approximate shape and size of the content it replaces.
Spinners only for in-place actions: button loading state, file upload progress.
Page-level data loads in sections — above-the-fold content loads first.
Minimum skeleton display time: 300ms (prevents flash for fast connections).
8.5 Accessibility Standards
WCAG 2.1 Level AA compliance target.
All images have descriptive alt text. Icons used alone have aria-label.
Colour contrast ratio: minimum 4.5:1 for body text, 3:1 for large text and UI components.
Focus indicators: visible on all interactive elements (3px indigo ring offset 2px).
Form inputs have associated labels (not just placeholder). Errors linked via aria-describedby.
Modal: focus trapped inside while open. Escape key closes. aria-modal and role="dialog" set.
Toast notifications: role="status" aria-live="polite".
Tab order: logical left-to-right, top-to-bottom. No positive tabindex values.

13. Security & Compliance
13.1 Authentication & Authorisation
All API routes require authentication except explicitly marked PUBLIC endpoints.
Role-based access control (RBAC): student, company, admin. Middleware validates role on each route.
Resource ownership: students can only modify their own data. Companies can only modify their own roles/company profile.
JWT access tokens: 15-minute expiry. Refresh tokens: 7-day expiry, stored in httpOnly cookie.
OAuth tokens: never stored server-side. Exchange for internal JWT immediately.
13.2 Data Protection
HTTPS enforced everywhere. HSTS header set. No HTTP → HTTPS redirect (HSTS-only).
Passwords: bcrypt with cost factor 12. Never stored in plain text or reversible encryption.
PII data (email, phone): encrypted at rest in database using field-level encryption.
File uploads: validated server-side for type and size. Virus-scanned before storage.
SQL injection: parameterised queries only. ORM (Prisma/Drizzle) used — no raw string queries.
XSS: all user input sanitised before storage. Content-Security-Policy header set.
CSRF: CSRF tokens on all state-changing operations. SameSite=Strict cookies.
13.3 Rate Limiting
Authentication endpoints: 5 requests/15 minutes per IP.
Application submission: 10 applications/hour per student.
Search API: 100 requests/minute per authenticated user.
File upload: 20 uploads/hour per user.
AI match computation: 50 requests/hour per user (scores cached to reduce load).
13.4 Compliance
India IT Act 2000 & IT (Amendment) Act 2008 compliance.
Data localisation: all Indian user data stored on servers within India (AWS ap-south-1 / Mumbai).
GDPR compliance for EU users (user data deletion, export, consent management).
Privacy Policy and Terms of Service must be reviewed by legal counsel before launch.
Age verification: users must confirm they are 18+ during signup.
Payment processing: PCI-DSS compliance via Razorpay/Stripe (never store card data ourselves).
